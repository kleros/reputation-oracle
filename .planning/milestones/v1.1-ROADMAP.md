## Overview

Production hardening milestone for the Kleros Reputation Oracle bot. Three phases turned the v1.0 PoC into a deployable, observable, failure-tolerant process: structured logging, transaction safety, and IPFS evidence upload. Started 2026-03-30, shipped 2026-04-22 (24 days).

## Milestone Summary

**Key Decisions:**

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| pino v10 over winston/bunyan | ESM-native, fastest JSON logger, callback flush for reliable exit | ✓ Phase 4 — NDJSON works with jq/fluent-bit without transform |
| Native fetch over Pinata SDK | 230+ transitive deps for one HTTP call; native fetch suffices | ✓ Phase 6 — zero new deps for IPFS upload |
| Skip-on-failure (no data-URI fallback) | Cleaner failure semantics; orphanedCids list surfaces manual unpin candidates | ✓ Phase 6 — user decision honored, batch isolation verified |
| Prepare/execute split in executeActions() | Guarantees all CIDs known before any tx; enables systemic-failure early exit | ✓ Phase 6 — 3-consecutive-failure escalation prevents wasted gas |
| `writeContract` never retried | Prevents duplicate-tx scenarios across nonce gaps | ✓ Phase 5 — documented as CLAUDE.md hardening pattern |
| Nonce increments after every mined tx (even reverts) | A reverted on-chain tx consumes a nonce | ✓ Phase 5 — CR-01 resolved, cascading nonce-too-low bugs eliminated |

**Issues Resolved:**

- Ambiguous tx state after RPC flakiness (null/timed-out receipts treated distinctly)
- Silent wallet depletion (balance preflight now exits before any tx attempt)
- SIGTERM leaving half-executed batches (signal handlers set flag, main loop checks)
- Secret leakage via console.error(err) on stack traces containing private keys (pino redact covers nested error causes)
- IPFS upload failures previously had no classification — 4-class taxonomy now drives retry behavior
- Evidence timestamp drift between gas estimate and submit (WR-01: build evidence once per action)

**Issues Deferred (documented as tech debt):**

- `06-SECURITY.md` threat-mitigation audit not run (security_enforcement=true gate bypassed at milestone close)
- `v1.1-MILESTONE-AUDIT.md` not run before close
- IN-01: dead `?? 30_000` fallback after zod default in `chain.ts` (advisory, no runtime impact)
- IN-02: `parseInt(disputeId)` precision loss above 2^53 (theoretical — Kleros dispute counts are orders of magnitude below this threshold)
- PROD-02: monitoring integration (health check / exit code reporting) — deferred from v1.1 scope
- PROD-03: key rotation documentation + Pausable contract upgrade — contract-level scope, future milestone

**Technical Debt Incurred:**

- Code-review fix-script uses Biome's default formatting; some edits landed with single-quote strings where double-quotes were used nearby (advisory-only, Biome enforces consistency on `lint:fix`)
- Integration test uses raw fetch rather than `uploadEvidenceToIPFS` — intentional to validate Pinata API contract independently, but duplicates the upload shape

**Test baseline:**
- 81 bot unit tests + 1 gated integration test (skipped without `PINATA_JWT`)
- `tsc --noEmit` clean
- 17 Foundry fork tests unchanged (v1.0 contract coverage)
